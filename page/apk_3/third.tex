\chapter{\threer}\label{three}

\section{What is it?}
The \textit{\threer} sample is a malicious Android application masquerading as a plug-in for a transportation app series developed by a South Korean developer. The legitimate series provides bus-related information, such as bus stop locations and arrival times, for the city of Jeonju, which makes the repackaged sample appear trustworthy at first glance. The package name \texttt{lmh.android.jjbus} and the internal name \texttt{jjbus} are consistent with a localised variant of the same Malbus family analysed in the previous chapters.

\subsection{Detection}
The antimalware analysis performed with VirusTotal shows that \texttt{42/78} security vendors flagged the file as malicious, while \texttt{29/78} vendors classified it as a secure file, \texttt{1/78} reported a timeout error, and \texttt{7/78} vendors reported an \textit{Unable to process file type} error.
Most engines identify the sample as a generic Android trojan, and some of them (e.g.\ CTX, Bitdefender, Arcabit) further classify it as a downloader-family specimen. A few vendors instead label it only as generic Android malware, without specifying a precise family.
It is also noteworthy that some well-known vendors, including Microsoft Defender, McAfee, and Malwarebytes, did not identify the file as malicious.

\section{Static Analysis}

Static analysis shows that the application combines legitimate bus-related functionality with advertising SDKs, remote configuration mechanisms, native code, and update-like behaviour. This indicates that the APK is not limited to its declared purpose, but may also support conditional execution and payload delivery at runtime.

\subsection{MainActivity.java}
\begin{itemize}
    \item Inherits from \texttt{BaseMainActivity}.
    \item Calls \texttt{getTabView(BaseTabFragmentActivity.MainTabs.BUSTRANS).setVisibility(8)} to hide a UI tab.
    \item Provides ad-network keys through \texttt{getAdvertiseKey(...)}.
    \item Retrieves configuration through \texttt{getEnvironmentProperties(...)}.
\end{itemize}

This combination suggests that the visible interface is at least partially controlled at runtime. In particular, the hiding of a UI tab and the retrieval of external configuration parameters imply that the application may adapt its behaviour according to remotely supplied values.
The use of multiple advertising providers, including AdMob, Firebase, and AdLib, further confirms the presence of monetization-oriented logic.

\subsection{EnvConstant.java}
The \texttt{EnvConstant} class defines several hardcoded advertising and analytics keys. MobSF confirmed the extraction of hardcoded secrets including \texttt{adpostkey}, \texttt{adlibkey}, encrypted strings, and hash keys embedded directly in the source code.

\begin{lstlisting}[language=Java]
public static final String ADAM_KEY = "3034Z1OT1385cf1db09";
public static final String ADLIB_KEY = "590ea88a0cf2843c93807ed9";
public static final String ADMOB_UNITID = "ca-app-pub-0388605861855854/7499990747";
public static final String FACEBOOK = "627046304133735685335228304842";
public static final String GA_TRACKER_ID = "UA-56668425-6";
\end{lstlisting}

These constants confirm the integration of multiple ad SDKs and tracking frameworks. Their presence in plaintext reduces transparency and can facilitate reverse engineering or unauthorised reuse of the embedded services.

\subsection{BusApiWeb.java}
\texttt{BusApiWeb} performs network requests to bus-related endpoints over plain HTTP. It uses Volley for networking, parses HTML responses instead of JSON, and relays information through listeners.
The exclusive use of HTTP means that bus-stop identifiers and route data travel in plaintext and can be intercepted or modified in transit, creating a clear man-in-the-middle risk.
The contacted domains include \texttt{jeonjuits.go.kr}, which is a legitimate Korean public transport API for the Jeonju city service.

The manifest declares 15 permissions, of which four are classified as dangerous: \texttt{ACCESS\_FINE\_LOCATION}, \texttt{ACCESS\_COARSE\_LOCATION}, \texttt{WRITE\_EXTERNAL\_STORAGE}, and \texttt{READ\_EXTERNAL\_STORAGE}. Nine permissions are identified as commonly abused by malware, including \texttt{INTERNET}, \texttt{WAKE\_LOCK}, and location-related permissions.\\
The app also declares 26 Activity components, 10 Services, 7 Receivers, and 1 Provider, of which 3 Activities, 3 Services, and 4 Receivers are exported and potentially accessible to other applications.
The presence of \texttt{REQUEST\_INSTALL\_PACKAGES}, GCM/Firebase messaging support, and multiple advertising activities is consistent with the patterns observed in the previous chapters.

\subsection{AppInfoActivity.java}
The \texttt{AppInfoActivity} displays a public bus-related URL and mainly serves to legitimise the app from a user-facing perspective.
It contains no real interaction logic, which makes it essentially cosmetic. Its role appears to be presenting the application as a legitimate transportation service, even though the rest of the static analysis reveals a much more complex and suspicious architecture.

\section{MobSF}

\begin{figure}[h]
    \centering
    \includegraphics[width=\linewidth]{mobsf_three_scorecard.png}
    \caption{MobSF Scorecard \threer}
    \label{fig:mobsf_scorecard_three}
\end{figure}

MobSF assigned a security score of \textbf{38/100} (grade C, HIGH RISK) to this APK, confirming that the application exposes a broad attack surface. The scorecard highlights problems related to signature schemes, outdated Android compatibility, exported components, insecure storage, hardcoded secrets, and the presence of 8 privacy trackers out of 432 checked.

\subsection{Certificate Analysis}
During the MobSF analysis, one unique certificate was found. The binary is signed with the \texttt{v1} signature scheme only, while \texttt{v2}, \texttt{v3}, and \texttt{v4} are absent. The certificate uses \texttt{SHA1withRSA}, which is considered weak due to known hash-collision vulnerabilities. This signing configuration relies on legacy mechanisms associated with known cryptographic and platform-level weaknesses.

\subsection{High Severity Findings}
MobSF reports multiple high-severity issues. The first is the vulnerability to the Janus attack, because the application is signed only with the v1 scheme, making it exploitable on Android 5.0--8.0. The second concerns SHA-1, which is vulnerable to hash-collision attacks. The third is the low minimum SDK version (\texttt{minSdk=14}), which allows installation on older Android versions with unresolved vulnerabilities.\\
The code analysis additionally identifies three high-severity issues: logging of sensitive information, world-readable files or SharedPreferences, and an active debug configuration. MobSF also detects the StrandHogg~2.0 weakness in the Google app-invite preview activity.\\
Finally, the shared library analysis reveals that \texttt{libAudio3.0.so} (compiled for multiple architectures) lacks both stack canary protection and fortify function hardening, making it susceptible to memory corruption attacks.

\subsection{Medium Severity Findings}
The medium-severity findings are mainly related to insecure configuration and component exposure. The flag \texttt{android:allowBackup=true} makes application data accessible through ADB backups when USB debugging is enabled.\\
Three Activity, three Service, and four Receiver components are exported without adequate protection, including components related to Firebase, GCM, and advertising frameworks. MobSF also warns about hardcoded secrets and API keys (\texttt{adpostkey}, \texttt{adlibkey}) found across multiple source files.\\
The application can read and write external storage, exposes insecure WebView usage, and uses MD5, which is considered weak due to collision-resistance problems.

\subsection{Low Severity Findings}
MobSF reports informational issues related to application logging, including potentially sensitive data. It also reports the use of SSL communication and the presence of root detection capabilities as secure findings.

\subsection{Overall Assessment}
Overall, MobSF confirms that the APK is not a simple transportation application. The combination of a score of 38/100, weak signing, exported components, insecure native libraries, hardcoded secrets, and an extensive ad/tracking infrastructure suggests an application with high security risk and behaviour consistent with a trojanized or heavily repurposed app.

\section{Genymotion}

The dynamic analysis performed in Genymotion provided practical confirmation of several findings obtained during static inspection. The application interacts with remote endpoints, relies on advertising frameworks, and appears capable of adapting part of its behaviour through external configuration data rather than through local code changes.
This behaviour is relevant because it indicates that the app is not limited to a simple transportation interface. Instead, it combines user-facing functionality with background services, ad-related activity, and network-driven logic, which is consistent with a repackaged or trojanized application structure.
The emulator analysis therefore strengthens the overall assessment of the sample as suspicious. In combination with insecure HTTP traffic, exported components, hardcoded secrets, and multiple privacy trackers identified by other tools, the Genymotion observations support the conclusion that the APK presents a non-negligible security risk.

\section{Conclusion}

The combined results of VirusTotal, static analysis, MobSF, and Genymotion consistently indicate that the analysed APK is suspicious and potentially malicious. Across the different tools, the same core patterns emerge: unencrypted network traffic, aggressive use of advertising libraries, remote configuration, exported components, and several weaknesses related to storage and cryptography.\\
The static inspection shows that the application is not limited to a simple bus-information interface. Instead, it includes native code, runtime permissions for package installation, hidden or manipulated UI behaviour, and remote update logic, all of which are typical indicators of repackaged or dropper-like software.\\
MobSF strengthens this conclusion by assigning a score of 38/100 and reporting high-severity issues including Janus vulnerability exposure, SHA-1 signing, StrandHogg weaknesses, world-readable files, debug configuration, hardcoded secrets, and unprotected native libraries. In parallel, the dynamic analysis in Genymotion confirms that the app interacts with remote services and relies on background components and advertising frameworks.\\
Overall, the APK should be treated as unsafe. Even if part of its visible functionality appears legitimate, the combination of insecure communication, suspicious permissions, and evasive implementation choices shows a clear mismatch between the declared purpose of the app and its actual security posture.
