\paragraph{What is it?} The \textit{\threer} sample is a malicious Android application masquerading as a plug-in for a transportation app series developed by a South Korean developer. The legitimate series provides bus-related information, such as bus stop locations and arrival times, for the city of Jeonju, which makes the repackaged sample appear trustworthy at first glance. The package name \texttt{lmh.android.jjbus} and the internal name \texttt{jjbus} are consistent with a localised variant of the same Malbus family analysed in the previous chapters.

\subsection{Detection}
The antimalware analysis performed with VirusTotal shows that \texttt{38/75} security vendors flagged the file as malicious, while \texttt{29/75} vendors classified it as a secure file, \texttt{1/75} reported a failure error, and \texttt{7/75} vendors reported an \textit{Unable to process file type} error.
Most engines identify the sample as a generic Android trojan, and some of them (e.g.\ CTX, Bitdefender, Arcabit) further classify it as a downloader-family specimen. A few vendors instead label it only as generic Android malware, without specifying a precise family.
It is also noteworthy that some well-known vendors, including Microsoft Defender, McAfee, and Malwarebytes, did not identify the file as malicious.

\subsection{Details}
Here is a detailed analysis of the permissions which are required by the APK, declared in the Manifest file (Fig.\ \ref{fig:manifest_third}):
\begin{enumerate}
\item \textbf{android.permission.ACCESS\_FINE\_LOCATION} $\rightarrow$ Allows the precise location of the device (GPS, mobile networks, Wi-Fi).
\item \textbf{android.permission.ACCESS\_COARSE\_LOCATION} $\rightarrow$ Provides approximate location (based on cellular towers and Wi-Fi).
\item \textbf{android.permission.ACCESS\_LOCATION\_EXTRA\_COMMANDS} $\rightarrow$ Allows additional commands to be sent to the location provider (rarely used in legitimate apps).
\item \textbf{android.permission.READ\_EXTERNAL\_STORAGE} $\rightarrow$ Allows reading files and data saved on external memory (photos, documents, downloads).
\item \textbf{android.permission.WRITE\_EXTERNAL\_STORAGE} $\rightarrow$ Allows you to create or edit files on external storage.
\item \textbf{com.android.browser.permission.WRITE\_HISTORY\_BOOKMARKS} $\rightarrow$ Allows editing of history and bookmarks in the stock Android browser.
\item \textbf{com.google.android.providers.gsf.permission.READ\_GSERVICES} $\rightarrow$ Reads Google Services Framework settings and data (e.g.\ device ID, GCM configurations).
\item \textbf{android.permission.INTERNET} $\rightarrow$ Enables any network connection (HTTP, sockets, DNS).
\item \textbf{android.permission.ACCESS\_NETWORK\_STATE} $\rightarrow$ Reads the status of network connections (Wi-Fi, mobile data).
\item \textbf{android.permission.ACCESS\_WIFI\_STATE} $\rightarrow$ Gets information about Wi-Fi networks (SSID, BSSID, status).
\item \textbf{android.permission.CHANGE\_WIFI\_STATE} $\rightarrow$ Changes Wi-Fi status (enable/disable, connects to networks).
\item \textbf{android.permission.WAKE\_LOCK} $\rightarrow$ Prevents the device from entering sleep mode.
\item \textbf{android.permission.VIBRATE} $\rightarrow$ Controls the vibration engine.
\item \textbf{lmh.android.jjbus.permission.C2D\_MESSAGE} $\rightarrow$ Enables reception of cloud-to-device messages (C2DM/GCM/FCM).
\item \textbf{com.google.android.c2dm.permission.RECEIVE} $\rightarrow$ Allows push notifications via Google Cloud Messaging (GCM/FCM).
\end{enumerate}

In addition to the permission analysis, VirusTotal assigns several behavioural and structural tags to the file: \texttt{detect-debug-environment}, \texttt{sets-process-name}, \texttt{checks-cpu-name}, \texttt{android}, and \texttt{apk}. The tags \texttt{detect-debug-environment}, \texttt{checks-cpu-name}, and \texttt{sets-process-name} are explicit anti-analysis and VM evasion indicators. The \texttt{detect-debug-environment} tag indicates that the application actively probes for debugger and sandbox contexts at runtime, \texttt{checks-cpu-name} reveals hardware fingerprinting consistent with virtual machine detection, and \texttt{sets-process-name} indicates process name manipulation to obfuscate the running process from monitoring tools.

VirusTotal shows the contacted domains, URLs, and IP addresses associated with the APK. The five contacted URLs are all served over plain HTTP, which means their content can be intercepted or modified in transit:

\begin{itemize}
    \item \texttt{http://busappmanager.elasticbeanstalk.com/v1/category/list.json}
    \item \texttt{http://busappmanager.elasticbeanstalk.com/v1/env/get.json}
    \item \texttt{http://busappmanager.elasticbeanstalk.com/v1/issue/my.json}
    \item \texttt{http://smartapps.cafe24.com/jjbus\_android/environment.json}
    \item \texttt{http://gwk.adlibr.com/ad/config/init}
\end{itemize}

The first three URLs are hosted on AWS Elastic Beanstalk (\texttt{busappmanager.elasticbeanstalk.com}) and appear to retrieve the app category list, runtime environment configuration, and remote issue data from a centralised backend. This mechanism is consistent with dropper-like behaviour, as it allows the app to modify its functionality without requiring a new APK release. The fourth URL (\texttt{smartapps.cafe24.com}) retrieves a configuration file specific to this \texttt{jjbus} variant, distinguishing it from the previous samples which contacted different regional \texttt{cafe24.com} endpoints. The domain \texttt{gwk.adlibr.com/ad/config/init} serves the advertising configuration and was already observed in the previous Malbus samples; it responds only to POST requests, otherwise returning \texttt{\{"result":4003,"message":"keymediakey"\}}.

The contacted domains are: \texttt{adlibr.com}, \texttt{busappmanager.elasticbeanstalk.com}, \texttt{cafe24.com}, \texttt{elasticbeanstalk.com}, \texttt{gwk.adlibr.com}, and \texttt{smartapps.cafe24.com}. No flagged domains were detected by VirusTotal at the time of analysis. No contacted IP addresses were reported directly in the Relations tab.

\subsection{Behavior}
VirusTotal's behaviour tab highlights several actions consistent with a suspicious application profile. The APK accesses file-related resources, builds JSON objects from filenames and absolute paths, reads data into buffer streams, and opens files from absolute paths. It also shows the use of reflection or control-based execution to start another application from the current one, which may indicate dynamic or indirect execution of external components. In addition, the app interacts with advertising and analytics libraries, scheduling recording tasks and reading files from the assets directory.

Notably, this variant carries explicit \textbf{anti-analysis and VM evasion} indicators not observed in the previous samples. VirusTotal tags the APK with \texttt{detect-debug-environment}, indicating that the application actively probes for debugger and sandbox contexts at runtime. The tag \texttt{checks-cpu-name} reveals hardware fingerprinting consistent with virtual machine detection, and \texttt{sets-process-name} indicates process name manipulation to obfuscate the running process from monitoring tools.

Overall, the observed behaviour suggests a complex application that performs file-system interaction, supports remote or dynamically driven actions, and is more evasion-aware than the previous Malbus variants. Additionally, a crowdsourced YARA rule match was recorded for this sample: the IcedID structural rule from the CAPEv2 repository (author: kevoreilly/threathive) matched the binary, consistent with the pattern observed across all three Malbus variants.



\section{Static Analysis}
The Java source code was decompiled using JD-GUI / Bytecodeviewer and inspected manually to identify classes and methods of interest.
Static analysis shows that the application combines legitimate bus-related functionality with advertising SDKs, remote configuration mechanisms, native code, and update-like behaviour. This indicates that the APK is not limited to its declared purpose, but may also support conditional execution and payload delivery at runtime.

\subsection{MainActivity.java}
\begin{itemize}
    \item Inherits from \texttt{BaseMainActivity}.
    \item Calls \texttt{getTabView(BaseTabFragmentActivity.MainTabs.BUSTRANS).setVisibility(8)} to hide a UI tab.
    \item Provides ad-network keys through \texttt{getAdvertiseKey(...)}.
    \item Retrieves configuration through \texttt{getEnvironmentProperties(...)}.
\end{itemize}

This combination suggests that the visible interface is at least partially controlled at runtime. In particular, the hiding of a UI tab and the retrieval of external configuration parameters imply that the application may adapt its behaviour according to remotely supplied values.
The use of multiple advertising providers, including AdMob, Firebase, and AdLib, further confirms the presence of monetization-oriented logic.

\subsection{BaseMainActivity.java}
\begin{itemize}
    \item Acts as the central initialization hub.
    \item Loads the native library with \texttt{System.loadLibrary("Audio3.0")}.
    \item Declares and invokes the native methods \texttt{startUpdate()} and \texttt{updateApplication()}.
    \item Requests the runtime permissions:
    \begin{itemize}
        \item \texttt{READ\_EXTERNAL\_STORAGE}
        \item \texttt{WRITE\_EXTERNAL\_STORAGE}
        \item \texttt{REQUEST\_INSTALL\_PACKAGES}
    \end{itemize}
    \item Loads remote app environment data, including a URL, a database version, and a remote download URL.
\end{itemize}

This class is the most relevant part of the static analysis. The presence of a native library, together with methods related to update handling and the request for \texttt{REQUEST\_INSTALL\_PACKAGES}, is consistent with dropper-like behaviour. In addition, the remote environment loading mechanism allows the application to modify its behaviour without requiring a new release.

\subsection{EnvConstant.java}
The \texttt{EnvConstant} class defines several hardcoded advertising and analytics keys. MobSF confirmed the extraction of hardcoded secrets including \texttt{adpostkey}, \texttt{adlibkey}, encrypted strings, and hash keys embedded directly in the source code.

\begin{lstlisting}[language=Java]
public static final String ADAM_KEY = "3034Z1OT1385cf1db09";
public static final String ADLIB_KEY = "590ea88a0cf2843c93807ed9";
public static final String ADMOB_UNITID = "ca-app-pub-0388605861855854/7499990747";
public static final String FACEBOOK = "627046304133735685335228304842";
public static final String GA_TRACKER_ID = "UA-56668425-6";
\end{lstlisting}

These constants confirm the integration of multiple ad SDKs and tracking frameworks. Their presence in plaintext reduces transparency and can facilitate reverse engineering or unauthorised reuse of the embedded services.

\subsection{BusApiWeb.java}
\texttt{BusApiWeb} performs network requests to bus-related endpoints over plain HTTP. It uses Volley for networking, parses HTML responses instead of JSON, and relays information through listeners.
\begin{itemize}
    \item Makes calls to \texttt{http://jeonjuits.go.kr} and related bus information endpoints.
    \item Uses Volley for networking.
    \item Parses HTML responses instead of JSON.
    \item Relays information through listeners.
\end{itemize}
The exclusive use of HTTP means that bus-stop identifiers and route data travel in plaintext and can be intercepted or modified in transit, creating a clear man-in-the-middle risk.

This activity appears to be mainly cosmetic. Its role seems to be presenting the application as a legitimate transportation service, even though the rest of the static analysis reveals a much more complex and suspicious architecture.

\section{MobSF}

\begin{figure}[h]
    \centering
    \includegraphics[width=\linewidth]{img/apk_3/mobsf_three_scorecard.png}
    \caption{MobSF Scorecard \threer}
    \label{fig:mobsf_scorecard_three}
\end{figure}

MobSF assigned a security score of \textbf{38/100} (grade C, HIGH RISK) to this APK, confirming that the application exposes a broad attack surface. The scorecard shows a high concentration of High and Medium severity issues, alongside informational and secure findings. Problems span six categories: certificate, manifest, code, trackers, secrets, and permissions, with 8 privacy trackers detected out of 432 checked and 4 permission hotspots flagged as critical.

\subsection{Certificate Analysis}
During the MobSF analysis, one unique certificate was found. The binary is signed with the \texttt{v1} signature scheme only, while \texttt{v2}, \texttt{v3}, and \texttt{v4} are absent. The certificate uses \texttt{SHA1withRSA}, with subject and issuer both identified as \texttt{C=ko, L=Jeonju, CN=Myoung-Hun Lee}. The validity interval spans from 2011-05-25 to 2061-05-12, and the reported hash algorithm is SHA-1. This signing configuration is weak from a security perspective because it relies on legacy mechanisms associated with known cryptographic and platform-level weaknesses, making it vulnerable to the Janus attack on Android 5.0--8.0.

\subsection{High Severity Findings}
MobSF reports multiple high-severity issues across certificate, manifest, and code categories. From the certificate, the application is vulnerable to the Janus attack due to v1-only signing, and SHA-1 is used as the hash algorithm. From the manifest, the low minimum SDK version (\texttt{minSdk=14}) allows installation on older unpatched Android versions, and the StrandHogg~2.0 weakness is present in the Google app-invite preview activity.\\
From the code analysis, three further high-severity findings emerge: logging of sensitive information, world-readable files or SharedPreferences, and an active debug configuration. Additionally, the code uses an insecure random number generator and weak cryptographic primitives (MD5), both of which reduce the reliability of any security-sensitive operations performed by the application.\\
Finally, the shared library analysis reveals that \texttt{libAudio3.0.so}, compiled for multiple architectures, lacks both stack canary protection and fortify function hardening, making it susceptible to memory corruption attacks.

\subsection{Medium Severity Findings}
The medium-severity findings are mainly related to insecure configuration, component exposure, and storage. The flag \texttt{android:allowBackup=true} makes application data accessible through ADB backups when USB debugging is enabled.\\
Three Activity, three Service, and four Receiver components are exported without adequate protection, including components related to Firebase, GCM, and advertising frameworks. MobSF also warns about hardcoded secrets and API keys (\texttt{adpostkey}, \texttt{adlibkey}) found across multiple source files, and flags insecure WebView usage.\\
On the storage side, files and SharedPreferences are world-readable, and the application can read and write external storage, potentially exposing user data to other applications. The use of MD5 for hashing is also flagged as a medium-severity issue due to its known collision-resistance weaknesses.

\subsection{Low Severity Findings}
MobSF reports informational issues related to application logging, which may include sensitive data. It also reports the use of SSL communication and the presence of root detection capabilities as secure findings. The scorecard additionally flags four permission hotspots as critical, reinforcing the idea that the application is highly sensitive from a privacy perspective.

\subsection{Overall Assessment}
Overall, MobSF confirms that the APK is not a simple transportation application. The combination of a score of 38/100, weak signing, exported components, insecure native libraries without memory protections, hardcoded secrets, world-readable storage, and an extensive ad/tracking infrastructure indicates an application with high security risk. The capability to silently install additional packages further widens the attack surface beyond what is expected from a bus-information service.

\section{Genymotion}

The dynamic analysis performed in Genymotion provided practical confirmation of several findings obtained during static inspection. The application interacts with remote endpoints, relies on advertising frameworks, and appears capable of adapting part of its behaviour through external configuration data rather than through local code changes.
This behaviour is relevant because it indicates that the app is not limited to a simple transportation interface. Instead, it combines user-facing functionality with background services, ad-related activity, and network-driven logic, which is consistent with a repackaged or trojanized application structure.
The emulator analysis therefore strengthens the overall assessment of the sample as suspicious. In combination with insecure HTTP traffic, exported components, hardcoded secrets, and multiple privacy trackers identified by other tools, the Genymotion observations support the conclusion that the APK presents a non-negligible security risk.

\section{Conclusion}

The combined results of VirusTotal, static analysis, MobSF, and Genymotion confirm that \threer\ is malicious and constitutes the most capable variant of the Malbus family analysed in this report.
The core dropper architecture, advertising SDK integration, and certificate weaknesses identified in the two earlier variants are fully preserved, but \threer\ introduces a qualitatively distinct expansion of the attack surface through the Crashlytics SDK, the unprotected native library \texttt{libAudio3.0.so}, and an elevated number of exported components.

The MobSF score of \textbf{38/100} (Grade~C, HIGH RISK) and the elevated count of high-severity findings reflect this expanded footprint: it is the only numerical risk rating explicitly produced across the three variants, underscoring the greater breadth of issues present in this sample.
The shared library \texttt{libAudio3.0.so}, compiled without stack canary or fortify function protections, represents an additional attack surface absent from the previous variants and makes this sample more susceptible to memory corruption exploitation.

The evasion layer further distinguishes \threer\ from \oner: the VirusTotal tags \texttt{detect-debug-environment}, \texttt{checks-cpu-name}, and \texttt{sets-process-name} confirm that sandbox and debugger detection, first introduced in \twor, are a stable feature of the later variants.
Together, these characteristics indicate that \threer\ was not merely maintained but actively developed to extend the malware's reach, making it the most forensically challenging sample in the Malbus family under analysis.
